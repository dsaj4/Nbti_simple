# 赛博判官报告分享页修改方案：舞台切换与自评/点评联动

状态：revision plan  
用途：沉淀当前讨论结果，作为下一步页面修改依据。本文只描述问题、目标、交互、文案和验收标准，不进入代码实现细节。

## 一、当前问题

### 1. 自评与点评的身份关系混乱

当前页面里，“罗永浩自评 / 苏格拉底自评 / 名人点评”这些说法容易混在一起。用户可能分不清：

- 是赛博判官这个类型在自我表达？
- 是名人在评价赛博判官？
- 还是名人也在认领自己属于这个类型？

这会削弱报告页的可信度和叙事清晰度。

### 2. 左侧舞台与右侧解释区存在双控制

之前右侧有名人点评 tab，左侧舞台也能切换名人。这样会形成两个控制中心：

```text
左侧切换舞台
右侧切换点评
```

用户可能会担心出现错配，例如：

```text
左侧是罗永浩图
右侧却是苏格拉底点评
```

这种错配会破坏“剧场演出”的整体感。

### 3. 判官回答与场景云的对应关系不够强

左侧上方场景云负责出题，下方判官回答负责给判词。但如果回答标签过于抽象，用户会觉得它只是随机关键词，无法理解“这是对上方场景的即时回应”。

### 4. 名人点评需要同时具备锋利和共鸣

如果名人点评只犀利，会像单纯嘲讽用户；如果只共情，又会失去赛博判官的锐评感。因此点评需要同时完成两件事：

- 点出赛博判官的要害。
- 让用户觉得“这不是在骂我，这是懂我”。

## 二、修改目标

本次修改的目标是把页面从“左右两个可切换模块”收束成一个完整的剧场叙事：

```text
开屏：赛博判官人物登场，自我发言
拉绳：舞台切换，名人演员登场
同步：右侧从赛博判官自评切换为名人点评
落点：左侧演出，右侧台词，二者形成同一幕
```

最终体验应当是：

> 用户一拉剧场绳，舞台切换，演员登场，右侧演员开始发表台词。

## 三、页面状态定义

页面存在三种主要状态。

### 状态 A：默认类型形象

左侧展示：

- 赛博判官类型人物。
- 上方场景云。
- 下方判官回答。

右侧展示：

- 模块标题：`赛博判官自评`
- 赛博判官以第一人称进行自评。
- 底部保留一句短判词。

### 状态 B：罗永浩名人幕

左侧展示：

- 罗永浩名场面图。
- 罗永浩幕场景云。
- 罗永浩幕判官回答。

右侧展示：

- 模块标题：`名人点评`
- 标题：`罗永浩说`
- 两段拟态口吻点评。
- 底部保留一句短判词。

### 状态 C：苏格拉底名人幕

左侧展示：

- 苏格拉底名场面图。
- 苏格拉底幕场景云。
- 苏格拉底幕判官回答。

右侧展示：

- 模块标题：`名人点评`
- 标题：`苏格拉底说`
- 两段拟态口吻点评。
- 底部保留一句短判词。

## 四、核心交互规则

### 1. 右侧不再提供名人 tab

右侧名人点评区删除独立 tab。名人点评不再由右侧自己切换，而是完全跟随左侧舞台状态变化。

### 2. 舞台切换控制整页叙事

用户通过左侧舞台控制进入不同状态。每次舞台切换时，下列内容必须同步变化：

```text
左侧名场面图
左侧场景云
左侧判官回答
右侧模块标题
右侧发言人标题
右侧两段正文
右侧短判词
```

不允许出现图片、回答和点评之间的错配。

### 3. 右侧点评区也需要动效

舞台切换时，右侧点评区要有同款“台词登场”动效。感受上不是静态内容替换，而是演员登台后开始发言。

推荐节奏：

```text
舞台转场
演员图出现
右侧模块标题变化
发言标题出现
第一段台词出现
第二段台词出现
短判词落下
```

动效不应太复杂，重点是建立“演员开始说话”的节奏。

### 4. 分享前可切换，分享后固定

用户在保存分享卡之前可以切换剧目和文案。  
一旦保存/分享，当前剧目、卡片和文案应当固化。

## 五、左侧舞台文案规则

左侧舞台仍然保持清晰的三段关系：

```text
上方：场景云 / 出题
中间：人物或名场面 / 演出
下方：判官回答 / 判词
```

判官回答要更口语化，像弹幕式即时反应。它必须直接回应上方场景云。

### 默认类型形象

场景云：

```text
一群人明明可以把问题说清楚，偏偏用热闹掩盖标准缺席。你脑内的小型审题庭，已经开始开庭。
```

判官回答：

```text
先审题
再锐评
标准不能软
```

### 罗永浩幕

场景云：

```text
理想主义发布会 vs 直播间的“交个朋友”。同一张嘴，两种真相。打脸归打脸，活儿还得干。
```

判官回答：

```text
打脸归打脸
系统更新中
活儿还得干
```

### 苏格拉底幕

场景云：

```text
雅典审判场，毒酒已备好。他不要赦免，只要定义。最后一口气，也留给追问。
```

判官回答：

```text
先问定义
概念过审
追问到底
```

## 六、右侧自评/点评文案规则

### 默认态：赛博判官自评

模块标题：

```text
赛博判官自评
```

正文方向：

- 使用赛博判官第一人称。
- 解释自己为什么不是爱吵架，而是受不了标准缺席。
- 语气要锋利，但要有自我辩护和价值感。

建议文案：

```text
“我不是爱吵架，我只是对‘差不多得了’生理性过敏。场面可以乱，逻辑不能乱。你要是问我为什么这么较真——因为问题一旦被糊弄过去，后面所有努力都在陪跑。”

“所以我先审题，再锐评。不是为了赢，是为了让标准重新站起来。”
```

短判词：

```text
赛博判官判词：场面可以乱，逻辑不能乱。
```

### 名人态：名人点评

模块标题：

```text
名人点评
```

发言标题规则：

```text
罗永浩说
苏格拉底说
```

正文分两段，但不显示小标题：

- 第一段：犀利、直接、点出赛博判官的要害。
- 第二段：参与式、感同身受、鼓励肯定。

两段都使用名人拟态口吻。

底部短判词保留，作为截图传播点。

## 七、名人点评文案

### 罗永浩说

第一段：

```text
“你这人最大的问题不是爱评价，是看见一锅浆糊还非要把标准捞出来。别人说差不多得了，你非要问差在哪、多在哪、凭什么得了。”
```

第二段：

```text
“但我理解你。人被现实打脸不可怕，可怕的是打完脸还不更新系统。你能把问题摆上桌，自己也一起审，这就不是嘴硬，这是还能继续干活。”
```

短判词：

```text
罗永浩同款判官：打脸是版本更新，不是卸载重装。
```

### 苏格拉底说

第一段：

```text
“你不是不会合群，你只是很难接受一个问题还没定义清楚，所有人就开始急着站队。你问一句‘这是什么意思’，他们就觉得你在抬杠。”
```

第二段：

```text
“但我理解你。没有定义，就没有真正的讨论。你不是在拖延答案，你是在保护问题不被糊弄过去。能把混乱问清楚，本身就是一种勇气。”
```

短判词：

```text
苏格拉底同款判官：死亡面前，先问定义，再谈情绪。
```

## 八、设计原则

### 1. 左侧负责演，右侧负责说

左侧是剧场演出：名场面、场景、判词。  
右侧是台词解释：自评或点评。

### 2. 默认是“我是谁”，切换后是“谁来点评我”

默认态回答：

```text
我是赛博判官，我为什么这样。
```

名人态回答：

```text
这个名人怎么看赛博判官。
```

### 3. 自评和点评不再混用

只有默认态叫自评：

```text
赛博判官自评
```

只有名人态叫点评：

```text
名人点评
```

### 4. 名人点评是拟态发言，不是真实背书

文案可以像名人在说话，但整体语境必须是剧场化参演，不能暗示名人真实授权、真实评价或真实参与。

## 九、验收标准

完成修改后，应满足以下条件：

- 开屏默认展示赛博判官类型形象。
- 开屏右侧标题为 `赛博判官自评`。
- 拉绳或切换舞台后，左侧进入对应名人幕。
- 名人幕右侧标题变为 `名人点评`。
- 右侧不再出现名人点评 tab。
- 右侧发言标题为 `罗永浩说` 或 `苏格拉底说`。
- 名人图、场景云、判官回答、右侧点评必须同步切换。
- 判官回答更口语化，并且能直接回应场景云。
- 每个名人点评有两段拟态口吻正文，不展示“犀利点评 / 参与式点评”小标题。
- 每个名人点评保留底部短判词。
- 右侧点评切换有轻动效，像演员登场后发表台词。
- 保存分享卡前可切换，分享后固化当前状态。
- 页面视觉变更后必须更新截图。

## 十、后续实现注意事项

本方案是设计与产品层面的修改依据。实际实现时要注意：

- 不要新增右侧独立切换入口。
- 不要让右侧点评和左侧舞台可以分开控制。
- 不要把名人点评写成名人真实背书。
- 不要让切换导致卡片高度明显跳动。
- 需要保持页面截图传播时的信息完整度。

## 十一、左侧舞台详细设计方案

### 1. 通用模型

为保证后续非判官型人格也能复用，左侧舞台不再绑定“审判 / 落槌”这类强判官概念，而统一抽象为：

```text
场景云 = 出题
演员 = 演出
回答气泡 = 说台词
```

这套模型可以服务所有人格类型：

- 对赛博判官：场景云抛出混乱问题，演员以锐评方式回应。
- 对治愈型人格：场景云抛出情绪问题，演员以安抚方式回应。
- 对社交型人格：场景云抛出关系问题，演员以调动气氛方式回应。
- 对行动型人格：场景云抛出卡住的问题，演员以推进方式回应。

因此本次左侧舞台的重点不是“法庭装饰”，而是让用户清楚看到：

```text
这个角色遇到了什么场景？
谁在登场？
他/她会怎么说？
```

### 2. 推荐结构

左侧舞台建议固定为三段：

```text
顶部：场景云
中部：演员区
底部：回答气泡
```

#### 顶部：场景云

场景云负责“出题”。内容格式固定为：

```text
几句描述 + 一个问句
```

推荐节奏：

```text
第 1 句：交代场景
第 2 句：制造冲突
第 3 句：强化反差
最后：抛出问题
```

它不负责解释人格，只负责把舞台情境说清楚。

#### 中部：演员区

演员区负责“演出”。默认展示类型人物，切换后展示名人名场面图。

演员区应成为左侧视觉中心，避免被过多文字、按钮或标签抢走注意力。

#### 底部：回答气泡

回答气泡负责“说台词”。它不是标签，而是角色对场景云的回答。

格式固定为：

```text
三句短句
每句带引号
下方居中排列
```

回答气泡放在演员下方居中，而不是贴近嘴边。这样更适配横图、单人图、多人图和后续 16 人格不同资产。

### 3. 三个状态的左侧文案

#### 默认类型形象

场景云：

```text
大家都说差不多得了。
方案看起来热闹，标准却还没站起来。
场面已经糊成一锅粥。

你怎么看？
```

回答气泡：

```text
“先审题。”
“再锐评。”
“标准不能软。”
```

#### 罗永浩幕

场景云：

```text
理想主义发布会还在回响。
直播间的灯也已经亮起。
以前说过的话，撞上了现在要做的事。

这算打脸，还是系统更新？
```

回答气泡：

```text
“打脸归打脸。”
“系统更新中。”
“活儿还得干。”
```

#### 苏格拉底幕

场景云：

```text
雅典审判场已经坐满。
毒酒也被端上来了。
所有人都等着他低头，他却还在追问概念。

死到临头，还要先审题吗？
```

回答气泡：

```text
“先问定义。”
“概念过审。”
“追问到底。”
```

### 4. 左侧设计重点

本次左侧修改需要解决三个表现力问题：

#### 关系清晰

用户必须能一眼看出：

```text
场景云在问
演员在演
回答气泡在说
```

#### 戏剧性更强

场景云不应像普通说明文字，回答气泡也不应像功能按钮。它们要共同构成“这一幕正在发生”的感觉。

#### 通用性更强

不要只为赛博判官写死“判官 / 审判 / 落槌”。可以保留赛博判官自身风格，但结构语言要能适配其他类型。

## 十二、GSAP 动效候选与执行计划

### 1. 总体动效叙事

动效不只是装饰，而是要服务这一条体验链：

```text
出题
演员登场
演员开口
右侧台词展开
```

推荐整体时序：

```text
1. 拉绳触发
2. 舞台短暂暗场
3. 旧场景云退场
4. 旧演员退场
5. 新场景云出现
6. 新演员登场
7. 回答气泡逐句冒出
8. 右侧自评/点评台词登场
```

### 2. 场景云动效

目标：让场景云像“题目被抛到舞台上”。

候选效果：

```text
从上方或侧上方滑入
轻微缩放回弹
描述句逐行出现
最后问句重点出现
```

推荐 GSAP 思路：

```text
cloud: opacity 0 -> 1, y -24 -> 0, scale .96 -> 1
lines: opacity 0 -> 1, y 8 -> 0, stagger 0.08
question: scale .98 -> 1.02 -> 1, color/highlight change
```

问句是场景云的发力点，可以轻微加粗、变色或做短促放大，让用户知道“演员要回答这个问题”。

### 3. 演员登场动效

目标：让中部人物/名场面成为真正的舞台演员。

候选效果：

```text
舞台灯光压暗
演员从幕布后淡入
聚光灯扫过
演员轻微放大定格
```

默认类型形象可更像角色亮相：

```text
scale .92 -> 1
opacity 0 -> 1
filter brightness slightly up
```

名人幕可更像剧照切换：

```text
old image opacity 1 -> 0, x -16
new image opacity 0 -> 1, x 16 -> 0
stage light sweep
```

### 4. 回答气泡动效

目标：让回答气泡像演员说出的三句台词，而不是标签。

固定位置：

```text
演员下方居中
```

候选效果：

```text
三句短台词逐个冒出
轻微弹跳
最终稳定排列
```

推荐 GSAP 思路：

```text
bubbles: opacity 0 -> 1, y 12 -> 0, scale .84 -> 1
stagger: 0.12
ease: back.out(1.5)
```

回答气泡的出现时间应晚于演员登场，形成：

```text
演员被点名
演员开口
```

### 5. 右侧自评/点评动效

目标：让右侧内容像演员登场后开始发表台词。

右侧内容不再独立切换，而是跟随舞台状态。

推荐节奏：

```text
右侧模块标题出现
发言人标题出现
第一段台词出现
第二段台词出现
短判词落下
```

推荐 GSAP 思路：

```text
section: opacity .65 -> 1, y 8 -> 0
title: opacity 0 -> 1, y 10 -> 0
paragraphs: opacity 0 -> 1, y 12 -> 0, stagger .1
quote: opacity 0 -> 1, scale .98 -> 1
```

右侧动效必须比左侧稍晚，避免用户感觉两个区域抢戏。

### 6. 拉绳切幕动效

目标：让用户感到“我触发了剧场换幕”。

候选效果：

```text
拉绳下拉
舞台轻晃
短暂暗场
幕布微动
新幕出现
```

推荐时长：

```text
总时长 900ms - 1400ms
```

过长会拖慢体验，过短会失去剧场感。

### 7. 动效降级

如果设备性能较弱或用户开启减少动态效果，应保留静态切换：

```text
内容即时替换
只保留轻微 fade
不执行聚光灯、回弹、弹幕类效果
```

## 十三、详细实施计划

### 阶段 1：结构重构

目标：先让“场景云 / 演员 / 回答气泡”关系清楚。

任务：

- 将左侧舞台调整为三段式表达。
- 场景云改为多行描述 + 问句。
- 回答从标签改为带引号短句气泡。
- 回答气泡固定在演员下方居中。
- 保留默认类型形象、罗永浩幕、苏格拉底幕三种状态。

验收：

- 不看右侧也能理解左侧发生了什么。
- 用户能分辨“上方在出题，下方在回答”。
- 回答气泡不再像按钮或标签。

### 阶段 2：状态联动

目标：确保整页只有一个剧目状态。

任务：

- 删除右侧名人点评 tab。
- 舞台切换时同步更新右侧自评/点评。
- 默认态右侧显示 `赛博判官自评`。
- 名人态右侧显示 `名人点评`。

- 苏格拉底幕显示 `苏格拉底说`。

验收：

- 不会出现罗永浩图配苏格拉底点评。
- 切换舞台时右侧内容同步变化。
- 用户只需要操作左侧舞台即可理解整页状态。

### 阶段 3：文案替换

目标：让左侧和右侧文案各司其职。

任务：

- 左侧场景云替换为“几句描述 + 一个问句”。
- 左侧回答气泡替换为带引号短句。
- 右侧默认态替换为赛博判官自评。
- 右侧名人态替换为两段拟态点评。
- 保留底部短判词。

验收：

- 左侧文案负责“演”。
- 右侧文案负责“说透”。
- 名人态第一段锋利，第二段共鸣。
- 不出现“犀利点评 / 参与式点评”小标题。

### 阶段 4：GSAP 动效接入

目标：增强舞台切换和台词登场表现力。

任务：

- 引入 GSAP 动效能力。
- 建立统一 timeline。
- 场景云逐行入场。
- 演员登场或切幕。
- 回答气泡 stagger 冒出。
- 右侧自评/点评同步入场。
- 拉绳触发舞台切换动效。

验收：

- 动效能清楚表达“出题 → 演出 → 说台词”。
- 动效不影响阅读。
- 切换时高度不明显跳动。
- 减少动态模式下仍可正常使用。

## 十四、推荐实现顺序

推荐不要先做动效。先保证结构和文案成立，再加 GSAP。

建议顺序：

```text
1. 左侧结构改为三段式
2. 替换三种状态文案
3. 删除右侧 tab，改为舞台控制右侧
4. 补齐状态数据模型
5. 接入基础 fade/slide 动效
6. 再加入 GSAP timeline
7. 生成截图验证
8. 推送 GitHub
```

这样可以避免在结构还没理清时就被动效复杂度拖住。

## 十五、无人值守长 goal 执行协议

本节用于支持后续无人值守或自动续跑式修改。执行 agent 不需要再向用户确认本方案中的已定事项，除非发现当前页面状态与本文档发生明显冲突。

### 1. 已定事项，不再反复询问

- 左侧舞台模型固定为 `场景云 = 出题 / 演员 = 演出 / 回答气泡 = 说台词`。
- 场景云使用 `几句描述 + 一个问句`。
- 回答气泡使用 `短句带引号`。
- 回答气泡固定在演员下方居中。
- 默认态右侧为 `赛博判官自评`。
- 名人态右侧为 `名人点评`。
- 右侧删除名人 tab，完全由左侧舞台切换控制。
- 切换时左侧图、场景云、回答气泡、右侧标题、正文和短判词必须同步。
- 每次视觉变更必须更新页面截图。

### 2. 实现优先级

无人值守执行时按以下优先级推进：

```text
P0：状态同步正确，不出现错配。
P1：左侧三段式关系清晰。
P2：右侧自评/点评文案正确。
P3：基础动效可用且不影响阅读。
P4：GSAP 增强动效更有剧场感。
P5：移动端和截图效果继续打磨。
```

如果时间不足，至少完成 P0-P2，并保留基础动效占位。

### 3. 允许的降级

如果 GSAP 外部资源无法加载，必须保留 CSS / Web Animations 风格的基础入场动效。

允许降级为：

```text

演员淡入
回答气泡逐个弹入
右侧台词淡入
```

不允许降级为：

```text
名人图和右侧点评不同步
右侧 tab 仍然存在
回答气泡变回无引号标签
场景云变回单句说明
```
